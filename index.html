01

Project Overview

NeoBank is a fully deployed, cloud-native digital banking backend built from scratch on AWS. The system enables users to create accounts, undergo AI-powered identity verification, transfer money in real time, and receive instant notifications — all backed by a secure, scalable serverless architecture.

The project is split into two main components: a monolithic Spring Boot application running on EC2 that handles authentication, user management, accounts, and KYC flows; and a suite of 8 specialized AWS Lambda functions that process transactions, detect fraud, write to the ledger, send notifications, and power an AI chatbot — all triggered by events rather than direct calls.

Spring Boot Backend

30+ REST endpoints covering authentication, user profiles, account management, and KYC document verification — deployed as a systemd service on an AWS EC2 t3.micro running Amazon Linux 2023.

Lambda Microservices

8 independently deployable Lambda functions triggered by SQS queues, S3 events, and API Gateway HTTP calls — processing every transaction through fraud detection, ledger writing, analytics, and notifications in parallel.

Core Application Flows

Money Transfer (ACID Transaction)

  1. Mobile app sends POST /prod/transactions with from_account, to_account, amount
  2. API Gateway routes to transaction-service Lambda
  3. PostgreSQL: SELECT FOR UPDATE locks sender's row — prevents race conditions
  4. Balance check → if insufficient funds, ROLLBACK and return error
  5. PostgreSQL: deduct from sender, add to receiver, COMMIT
  6. Event published to SQS queue → triggers fraud-checker, ledger-writer, notification-service, analytics-processor in parallel

06

Lambda Functions

transaction-service

API Gateway POST

Role: Financial core. Handles every transfer with PostgreSQL row locking and ACID transactions.

fraud-checker

SQS Trigger

Role: Real-time fraud detection using risk scoring (amount, time, weekend, round numbers). Scores >0.75 trigger alerts.

ledger-writer

SQS Trigger

Role: Immutable transaction ledger to DynamoDB.

kyc-validator

S3 Event Trigger

Role: AI identity verification with Rekognition, updates PostgreSQL, sends SNS notification.

07

Database Design

PostgreSQL RDS: users, accounts, user_sessions, kyc_documents, contacts, user_settings — all ACID-compliant.

DynamoDB: transactions table with GSI from_account-timestamp-index for fast account history queries.

08

Security Design

Passwords stored in Cognito (never in PostgreSQL). JWT validation via Spring Security filter. SELECT FOR UPDATE row locking prevents double-spending. Freeze requires password re-verification. Secrets injected via systemd EnvironmentFile.

Security Note: IAM least-privilege roles. No secrets in source code. Cognito brute-force protection (5 attempts → 15min lockout).
